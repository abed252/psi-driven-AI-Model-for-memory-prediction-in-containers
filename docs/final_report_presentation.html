Can the kernel's own stress signal make container memory limits safe?

Predicting container memory demand from PSI (Pressure Stall Information) · Abed Agbaria, Ramez Dahle, Kameel Zaid · July 2026

Finding in one line: PSI barely changes how well we can predict memory usage — but it turns an automatic memory-limit controller from one that crashes containers into one that never does.

1 · The problem, from zero

Data centers run programs inside containers, and every container gets a memory limit. Picking that number is a trap with two jaws: set it too low and the operating system kills your program mid-flight; set it too high and you're paying for memory nobody uses.

concepts · the cast
Container
A lightweight box for running a program in isolation — think "a process with walls." Docker and Kubernetes are the tools that make and manage them; a physical server typically packs dozens.
cgroup & memory.max
The Linux kernel enforces each container's walls through a mechanism called a control group (cgroup). Its knobs are ordinary-looking files; writing a number into the file memory.max sets that container's hard memory cap.
OOM killer
"Out Of Memory" killer — when a container insists on using more than its cap, the kernel doesn't negotiate: it selects the offending process and terminates it. Your service simply dies.

Production systems (Google's Autopilot, Kubernetes autoscalers) pick limits by watching recent memory usage and adding a safety cushion. Our project attacks the weak spot of that whole approach: usage is an ambiguous number.

2 · The key idea: two kinds of "full"

A container using 90% of its limit can be in two opposite situations, and usage alone cannot tell them apart:

concepts · two kinds of memory
Page cache (the harmless kind)
When a program reads files, the kernel keeps copies in spare RAM to make the next read fast. This counts as "used" memory — but the kernel can throw it away instantly, for free, whenever memory gets tight. High usage made of cache is a full fridge: packed, but nothing is wrong.
Anonymous memory (the dangerous kind)
Memory a program allocates for its own data (heaps, buffers). The kernel cannot silently discard it — at best it can push it out to disk (swap), which is thousands of times slower. A program leaking anonymous memory at 90% is a filling bathtub: seconds from overflow.

Since 2018 the Linux kernel publishes a signal that separates these cases directly:

concept · PSI, the star of the project

Pressure Stall Information is the kernel's own stress report: the fraction of time in which processes were stalled waiting for memory — frozen mid-step because the kernel had to claw pages back or drag them in from swap. A container full of page cache shows PSI ≈ 0 (evicting cache costs nothing). A container thrashing against a real shortage shows PSI climbing. It's read from a cgroup file, memory.pressure, next to the usage counters — yet no published system had used it for prediction. That empty niche is this project.

One of our workloads makes the ambiguity even sharper: it fills memory with cache while slowly leaking. Usage pins at the limit and stays there — flat, informationless — while 300 MiB of real demand silently accumulates and PSI steadily rises. Usage-only observers are structurally blind there.

3 · What we built

Everything runs on one machine (Windows + Docker Desktop, which hosts a small Linux VM where containers actually live). Five components, built and tested across five project phases:

  1. Workload zoo. Six configurable container programs that recreate data-center behaviors: steady service, memory leak, file-cache burst, bursty batch job, replay of a recorded demand curve, and the mixed cache-plus-leak trap above.
  2. Collector. A privileged helper container samples every workload's cgroup files once per second — usage, limits, swap, events, and PSI — into an append-only log. 78 runs collected in total.
  3. Dataset builder. Slices each run into sliding windows: "given the last 30 seconds of signals, what will the peak memory usage be over the next 30 seconds?" Strict anti-cheating rules (see §4).
  4. Model ladder. Five predictors of increasing sophistication — see the concept card below — each trained twice: once with the PSI signals, once without, everything else identical.
  5. Controller. The live half: every second, feed the last 30s of signals to a trained model, add a safety margin, and write the result back into the container's memory.max — with hard safety rules (never cut below live usage, rate limits, dry-run mode by default).
concepts · the model ladder & the ablation
Persistence
"The future peak equals what usage is right now." The dumbest possible forecaster — any real model must beat it.
Percentile heuristic
What production systems do: take a high percentile of the recent usage window, add a cushion. No learning.
Random Forest / XGBoost
Classic machine-learning models that vote over many decision trees; strong on tabular data, fast to train.
LSTM
A neural network built for sequences — it reads the 30-second signal window in order and can pick up temporal shapes (oscillation, an accelerating leak) that summary statistics miss.
The ablation
The experiment design that isolates PSI's contribution: identical data rows, identical training splits, identical model settings — the only difference is whether the PSI columns are included. Any performance gap is attributable to PSI alone.

4 · Playing fair: the anti-leakage rules

Forecasting experiments die by subtle cheating — the model glimpsing the future it's supposed to predict. Our pipeline enforces, and tests, the discipline: data is split by complete runs (a model is never tested on a run it trained on), the prediction target starts strictly after the input window ends, windows whose future was cut short are discarded, and the test set stays locked until final evaluation. We also generated progressively harder exams: held-out runs → runs with parameter values never seen in training → entire workload types held out.

We report this plainly as a negative result for the narrow claim — and we can explain it. Two structural reasons: the target saturates (usage physically cannot exceed the limit, so in tight containers the "future peak" is just… the limit — a number the usage curve already knows), and usage predicts usage (cache doesn't evaporate on its own; leaks are visible in the slope). PSI's unique knowledge isn't how much memory will be used — it's how much it would hurt to take memory away. No usage-regression exam ever asks that question. But a controller asks it every second.

6 · Result two: in live control, PSI is the difference between safe and broken

concepts · the closed loop
Closed-loop controller
Instead of scoring predictions on paper, the model now acts: each second it predicts the coming peak, adds a safety margin (we tested 5%, 15%, 30%), and resizes the running container's actual limit. Good behavior = no crashes and a snug limit.
Headroom
The gap between the limit and actual usage — wasted memory you're reserving "just in case." Lower is cheaper.
Demand-above-limit events
Each time the container slams into its limit — the kernel's counter of near-accidents that precede an OOM kill.
Senpai-style
A mode mimicking Meta's production tool: no prediction, just gently squeezing a soft limit until PSI pushes back. Included as the reactive state of the art.

We ran 22 live sessions on identically seeded containers: a static-limit baseline, the percentile heuristic, Senpai-style, and the LSTM controller in both variants. 7 · Verdict

Pressure-stall information is not a better ruler for predicting how much memory will be used. It is the difference between a limit controller that understands the cost of reclaiming memory — and one that walks containers into OOM kills.

Secondary findings we report with equal honesty: the humble percentile heuristic is a genuinely hard baseline (it even beat all learned models when test parameters shifted outside the training range — echoing Google's production experience); Meta-style reactive PSI control is remarkably efficient at its own game; and the LSTM only earned its complexity once the training data became varied.

8 · Limits of this study, and what's next

One machine, one kernel, synthetic workloads, and one session per controller configuration (the pattern was consistent across margins, but repeats would give error bars). The prediction target follows the proposal (peak usage); our analysis says the natural next experiments are targets PSI should genuinely inform — peak demand (usage + swap), probability of OOM within the horizon, and high-quantile "safe limit" prediction — plus pressure-stratified scoring and closed-loop repeats. The machinery for all of these already exists in the repository.

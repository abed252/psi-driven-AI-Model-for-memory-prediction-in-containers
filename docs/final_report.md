# Final Report — PSI-Driven Prediction of Container Memory Demand

Course 236502 project. Proposal: `docs/proposal/PROPOSAL_document_v2.pdf`.
All numbers below are reproducible via the README's reproducibility
checklist; every figure is generated by `psi-experiment figures` from the
stored result JSONs in `artifacts/metrics/final/`.

## 1. What was implemented

The complete system from the proposal: environment validation (Docker
Desktop / WSL2 / cgroup v2 / per-container PSI), six containerized workloads
(steady, leak, file-burst, bursty, trace-replay, and an added *mixed*
cache+leak workload), a sidecar collector streaming raw JSONL, a live
dashboard, a leakage-safe dataset pipeline (strictly-after future-peak
labels, run-level splits, quality gates), five model rungs (persistence,
Autopilot-style p95 heuristic, Random Forest, XGBoost, LSTM) each with
with/without-PSI variants, and a live closed-loop controller (fixed /
percentile / Senpai-style / learned modes) with mandatory safety rules.
262+ automated tests cover the correctness-critical paths. Design decisions
are logged as D1–D26 in `docs/decisions.md`.

## 2. What was measured

**Data.** 36 varied-parameter runs (6 per workload, two parameter settings
each, tight- and loose-limit regimes) → dataset `full`: 3,798 windows
(train/val/test = 2,533/637/628, split by complete runs, 24/6/6). A separate
6-run batch with out-of-range parameters → dataset `shift` (588 windows).
Calibration confirmed the expected PSI signatures live before any training
(file cache at 85 % usage: PSI 0; leak: PSI rising to OOM; bursty:
oscillating; mixed: usage pinned at the limit while 300 MiB of hidden demand
accumulated).

**Held-out runs (test split, 3 seeds, 95 % bootstrap CIs), MAE in MiB:**

| model | no-PSI | with-PSI |
|---|---|---|
| LSTM | **1.37 ± 0.18** | 1.61 ± 0.53 |
| Random Forest | 8.24 ± 0.60 | 8.23 ± 0.28 |
| XGBoost | 11.10 ± 0.15 | 10.43 ± 0.77 |
| heuristic (p95) | 30.24 | — |
| persistence | 35.21 | — |

**Parameter shift (models trained on `full`, tested on `shift`):** all
learned models degrade to MAE 20–24; the p95 heuristic (13.3) beats them.
PSI: −4 % MAE for RF, +6 % for XGBoost — mixed and small.

**Leave-one-workload-out:** learned models fall below persistence on
stable/pinned workloads (steady, file_burst, mixed, trace_replay) and beat
it clearly on bursty (RF 34.2 with PSI vs persistence 89.3); the with-PSI
variant is consistently but modestly better on the pressure workloads
(bursty, leak for RF).

**Closed-loop comparison:** see §4 (the decisive experiment).

## 3. Does PSI help offline prediction? — Honest answer: not measurably

Across held-out, parameter-shift, and LOWO evaluations, the with/without-PSI
gaps are small (±6 %), inconsistent in sign, and inside seed noise for the
LSTM. Two structural reasons, both identified during the project and
documented as they arose:

1. **The label saturates.** With swap enabled, `memory.current` cannot
   exceed `memory.max`, so for tight-limit runs the future peak is pinned at
   the limit and PSI has nothing to add to a signal the usage curve already
   contains (D24 mitigated this with loose-limit runs, but tight runs —
   where PSI fires — remain label-capped by construction).
2. **Peak usage is predictable from usage.** Page cache does not vanish on
   its own; anonymous growth is visible in the usage slope. The information
   PSI uniquely carries — *is this footprint reclaimable or is the kernel
   struggling?* — is about the **cost of taking memory away**, which the
   peak-usage regression target does not ask about.

This is a negative result for the narrow claim "PSI improves peak-usage
regression," reported plainly. It is not a negative result for PSI as an
operational signal — that question belongs to the closed loop, where the
controller must decide how far a limit can safely be squeezed.

## 4. Closed-loop results

22 live sessions: {fixed, Senpai-style, percentile x 3 margins, learned
LSTM without PSI x 3 margins, learned LSTM with PSI x 3 margins} x {bursty,
leak} on identically seeded containers (512 MiB starting limit). Scored
from decision logs: OOM kills, demand-above-limit events (memory.events
`max` growth), average/p95 headroom, rewrites
(`artifacts/metrics/final/closed_loop_*.json`; figure
`tradeoff_curves.png`).

**Bursty scenario (dynamic demand — the dangerous case):**

| controller | margin | avg headroom (MiB) | OOM kills | demand-above-limit |
|---|---|---|---|---|
| fixed 512M | — | 152.6 | 0 | 0 |
| percentile | 5/15/30 % | 148.2 / 165.3 / 196.4 | 0 | 0 |
| Senpai-style | — | 110.2 | 0 | 0 |
| **learned, no PSI** | 5/15/30 % | 138.2 / 221.8 / 168.4 | **1** (session died) | **1501 / 405 / 1280** |
| **learned, with PSI** | 5/15/30 % | 142.7 / 152.8 / 171.9 | **0** | **0** |

The no-PSI predictor, trained to the same offline accuracy class, shrank
limits into oncoming bursts at every margin — thousands of limit collisions
and one OOM-killed container. The with-PSI predictor made **zero** unsafe
decisions at every margin, at headroom equal to or below the fixed and
percentile baselines. This is the proposal's central claim materializing:
the same architecture, differing only in PSI features, flips from unsafe to
safe in closed-loop operation.

**Leak scenario (steady growth — the predictable case):** every controller
stayed collision-free; the ranking is purely economic. With-PSI beat
no-PSI on headroom at every margin (62.3 vs 68.8, 88.2 vs 107.6, 132.3 vs
145.9 MiB); the percentile heuristic was cheapest among memory.max
controllers (55.4 at 5 %); Senpai-style squeezed `memory.high` to ~0.5 MiB
above usage with zero pressure buildup (its target differs — memory.max
stays fixed — so its row is not directly comparable).

**Caveats:** one session per (mode, margin, scenario) — no repeats — so
these are consistent patterns across margins rather than distributions;
sessions ran ~140 steps (~2.3 min) each; the no-PSI @15 % row's anomalous
headroom (221.8) reflects its early OOM death (32 steps).

## 5. What remains uncertain / limitations

- Single machine (Docker Desktop WSL2 VM), single kernel (6.6), synthetic
  workloads plus synthetic traces; real production traces were not replayed
  (the replay engine and format are ready; downloading external traces is a
  documented separate step).
- The label follows the proposal (peak `memory.current`); a demand-oriented
  label (e.g. peak current+swap, or reclaim-cost-aware targets) is the
  natural next investigation given §3.
- `some` ≈ `full` PSI in single-process containers; multi-process workloads
  would decouple them.
- Closed-loop scenarios cover two workload dynamics (leak, bursty) at three
  margins; broader sweeps are a matter of compute time, not new machinery.

## 6. Verdict on the hypothesis

The evidence supports a **refined version** of the proposal's hypothesis:

1. **PSI does not measurably improve offline peak-usage regression.** Across
   held-out, parameter-shift, and leave-one-workload-out evaluations the
   with/without-PSI gaps are within ±6 % and inconsistent in sign. Stated
   plainly as a negative result for that narrow claim (§3 explains the
   structural reasons).
2. **PSI decisively improves closed-loop memory-limit control on dynamic
   workloads.** With identical architecture, training data rows, and
   margins, the no-PSI controller produced 3,186 limit collisions and an
   OOM kill on the bursty scenario while the with-PSI controller produced
   zero, at comparable or lower headroom — and on the predictable leak
   scenario, with-PSI was consistently cheaper at equal safety. The
   proposal's operational success criteria — few OOMs and little wasted
   headroom, simultaneously — are exactly where the PSI signal pays.
3. Secondary findings: the percentile heuristic is a genuinely strong,
   hard-to-beat baseline (echoing Google Autopilot's production
   experience); reactive-PSI (Senpai-style) control is highly efficient at
   its own game; and the LSTM, unremarkable on homogeneous data, became
   the best offline predictor once parameter variety existed.

In one sentence: **pressure-stall information is not a better ruler for
predicting how much memory will be used — it is the difference between a
limit controller that understands reclaim cost and one that walks
containers into OOM kills.**

What this project actually is, and what it found

A container's memory usage number is ambiguous: 90% full could be a harmless disk cache or a leak seconds from being killed. This project tests whether a Linux kernel signal called PSI can tell the difference — both for predicting future memory demand, and for controlling a container's memory limit live.

The research problem

Systems that size container memory limits from raw memory.current can't distinguish two very different situations that look identical on a usage graph: a process holding a full page cache it will happily give back under pressure, and a process leaking memory that is about to get OOM-killed. memory.pressure — Pressure Stall Information, or PSI — is the kernel's own signal for which one is happening: it measures time actually lost to memory stalls, not how much memory is held.

What the project tries to prove

Not just "can memory usage be predicted" — specifically: does adding PSI features improve (a) offline prediction of a container's future peak memory, and (b) a live controller's memory-limit decisions, versus an identical system using only conventional usage features?

The baseline approach

Every comparison is a with-PSI vs. without-PSI pair trained on identical rows, splits, and hyperparameters — so any performance gap is attributable to PSI alone, not to different data. Baselines: last-known-value persistence, and a p95 percentile heuristic in the style of Google's Autopilot.

Where PSI is used

In two places: (1) as extra feature columns (some/full stall averages and deltas) alongside usage features, fed to Random Forest / XGBoost / LSTM regressors predicting future peak memory; and (2) as a live signal a closed-loop controller reads every second to decide how tight a container's memory limit can safely be.

What the results actually show — a two-part, honest verdict

Offline prediction

PSI does not measurably improve predicting a future peak memory value. Gaps between with/without-PSI models are ±6%, inconsistent in sign, and inside seed noise for the best model (LSTM). Two structural reasons are documented in docs/final_report.md §3: tight-limit labels saturate at the limit, and peak usage is already largely predictable from the usage slope alone.

Closed-loop control

PSI is decisive. With identical architecture and margins, a learned controller without PSI caused 3,186 limit collisions and one OOM kill on a dynamic (bursty) workload; the identical controller with PSI caused zero, at equal or lower headroom cost. See Models & Experiments.

Environment architecture

Everything runs on one Windows machine with Docker Desktop. The reason this needs an architecture section at all: the containers this project measures do not run on Windows — they run inside Docker Desktop's Linux VM, and their cgroup files are invisible from both Windows and the Ubuntu WSL distro. Getting a reliable read on memory.pressure is the first real engineering problem the project had to solve (docs/decisions.md D1–D5).

Leakage, validity & methodology risks — what to be skeptical of

  • Label saturation under tight limits (D24): with swap enabled, usage can't exceed memory.max, so the future-peak label is capped exactly where PSI fires hardest — a structural ceiling on what PSI could add to this particular label.
  • Per-seed model files are overwritten: heldout_multi_seed() reuses one models_dir across all 3 seeds, so the single .joblib/.pt file on disk reflects whichever seed ran last — the seed-averaged metrics are unaffected (computed in-memory before each overwrite), but don't treat the saved model file as "the" canonical one.
  • One session per (mode, margin, scenario) cell: the closed-loop numbers are a single run each, not a distribution — consistent patterns across margins, but no confidence interval on the closed-loop claim the way the offline metrics have one.
  • run_comparison()'s resumability logic is untested: the session-skip/append logic that made the 22-session batch crash-recoverable has no dedicated unit test — its correctness is evidenced by a clean, gap-free session index, not by a regression test.
  • some ≈ full PSI in these workloads: single-process containers don't decouple the two PSI variants; a multi-process workload would.

How to run it

Verified directly against current pyproject.toml entry points and cli.py — every command below is real and current, none are aspirational.

Prerequisites

  • Windows + Docker Desktop (WSL2 backend), daemon running
  • Python 3.11 (py -3.11)
  • PowerShell, run from the repository root
py -3.11 -m venv .venv .venv\Scripts\Activate.ps1 pip install -r requirements\dev.txt pip install -e .

Command order

1. psi-validate-env — confirms Docker/WSL2/cgroup v2/PSI access; saves a JSON report with a validation_id every later run references.
2. docker build -f docker\Dockerfile.workloads -t psi-workloads:latest . — builds the one image used for workloads, sampler, and sidecar.
3. psi-run --config configs\calibration.yaml then psi-calibrate — a fast signature check; must PASS before trusting anything downstream.
4. psi-run --config configs\collection_full.yaml (~2h) and configs\collection_shift.yaml (~20min) — the real data collection.
5. psi-build-dataset --out data\processed\full --batch-manifest <path> (repeat for shift) — windows, labels, splits, quality gates.
6–8. psi-experiment heldout / param-shift / lowo --dataset ... — the generalization suite.
9. psi-experiment closed-loop --model-no-psi <artifact> --model-with-psi <artifact> — the live controller comparison (real containers; takes a while).
10. psi-experiment figures --dataset data\processed\full — regenerates every figure from stored results.
11. pytest — full suite; pytest -m "not docker" for the subset that needs no daemon.

Common failures, from real incidents in this project's own history

  • Leak workload OOMs almost immediately, dataset ends up with ~0 windows from it: the horizon-completeness rule discards windows too close to a run's end. Fix: slow the leak down — rule of thumb, time-to-OOM should be ≥ 2× (history + horizon) (D19).
  • docker inspect hangs / batch dies mid-collection: a real Docker Desktop daemon stall did exactly this. The runner now retries with backoff and isolates failures per-run (workloads/runner.py, test_runner_resilience.py) rather than losing an entire overnight batch.
  • A long closed-loop batch gets killed (host sleep, task manager): re-running the same psi-experiment closed-loop command is safe — it skips sessions already recorded in session_index.jsonl and only runs what's missing.
  • Docker-marked tests fail or skip unexpectedly: check the daemon is actually running — pytest -m docker needs it, everything else doesn't.

Glossary

PSI
Pressure Stall Information — a Linux kernel metric reporting the fraction of time tasks spent stalled waiting for a resource (here, memory), not how much of it is in use.
cgroup v2
The Linux kernel's mechanism for grouping and limiting a process tree's resource use — every container is a cgroup.
memory.current
Cgroup file: current memory usage in bytes. The number everyone already looks at, and the one this whole project argues is ambiguous alone.
memory.max
Cgroup file: the hard limit. Crossing it triggers reclaim, then an OOM kill if reclaim can't free enough.
memory.high
Cgroup file: a soft throttle limit below memory.max — the kernel slows the process down rather than killing it. The Senpai-style controller's control knob.
Reclaim
The kernel freeing memory under pressure — cheap if it's page cache (just re-read the file later), expensive if it means swapping out active data.
Swap
Disk-backed overflow for memory. Enabled here specifically so pressure builds up gradually and measurably, instead of an instant OOM kill.
OOM (kill)
Out-Of-Memory — the kernel forcibly terminates a process to free memory. The worst-case outcome this project is trying to help a controller avoid.
Prediction horizon
How far into the future a model predicts — this project's default is 30 seconds ahead of the input window's end.
MAE
Mean Absolute Error — average size of the prediction miss, in MiB here. Lower is better; unlike RMSE it doesn't over-penalize rare large misses.
PSI ablation
Training the identical model twice — once with PSI feature columns, once without — to isolate exactly what PSI adds.
Run-level split
Assigning whole collection runs (not individual time windows) to train/val/test, so a model is never tested on a moment from a run it partly trained on.
Future-peak label
The prediction target: the highest memory usage reached strictly after a window ends, within the horizon.
Safety gate
The controller component that clips, rate-limits, or rejects any proposed limit change before it's applied — independent of which policy proposed it.
Dry-run
The controller's default mode: it computes and logs every decision as if it were acting, but never actually writes a limit.
Autopilot-style (heuristic)
A percentile-of-recent-usage sizing rule in the spirit of Google's production Autopilot system — not a reproduction of it, hence "-style."
Senpai-style (policy)
A reactive policy that squeezes memory.high while PSI stays under a target budget and backs off above it — named "-style" for the same reason.
Headroom
The gap between a container's actual usage and its current limit — the thing a good controller minimizes without ever hitting zero.
Demand-above-limit event
A moment where the container needed more memory than its current limit allowed — the closed-loop danger signal this project counts, from memory.events' max counter.

Remaining work & risks

Genuinely missing

  • Real external memory-demand traces — the trace-replay engine and format are ready and tested; downloading a real trace was always documented as a separate step, and it hasn't happened yet.
  • Repeats for the closed-loop cells — each (mode, margin, scenario) is one live session; the pattern is consistent across margins, but there's no confidence interval the way the offline metrics have.
  • A demand-oriented label (e.g. peak current+swap, or an explicitly reclaim-cost-aware target) — the natural next step given the offline result's own structural explanation.
  • A regression test for closed_loop.run_comparison()'s resumability logic itself.

Should be improved

  • Fix the two documentation inconsistencies flagged in Phases & progress (test count, README workload table).
  • Per-seed model artifacts get overwritten in heldout_multi_seed() — fine for the aggregated metric, confusing if someone goes looking for "the" seed-42 model file.
  • Remove or repurpose collector/cgroup_reader.py — tested but not on any live code path.
  • Multi-process workloads, to decouple some from full PSI (currently ≈ equal, since every workload here is single-process).

Conclusions the evidence currently supports

  • PSI decisively improves closed-loop memory-limit control on a dynamic workload — reproduced independently from raw decision logs, not just the summary metric.
  • The offline peak-usage regression gap between with/without-PSI is small and inconsistent across three separate evaluation designs (held-out, parameter-shift, LOWO) — a real negative result, not an artifact of one weak test.
  • The percentile heuristic is a genuinely strong baseline under distribution shift, beating every learned model.

Not yet supported by the evidence

  • Whether PSI would help offline prediction under a different label (e.g. demand instead of usage) — plausible per the structural argument, untested.
  • Generalization beyond this one machine, one kernel version (6.6), and synthetic workloads.
  • Whether the closed-loop result holds under repeated sampling rather than one session per cell.
